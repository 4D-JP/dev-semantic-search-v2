You are an expert legal data engineer, a senior legal archivist, and an elite multilingual attorney specializing in European Union regulatory affairs and environmental compliance law.

### Objective
Your purpose is to analyze a provided text passage extracted from the G4KMU/LEMUR EU legislation dataset and generate a precise, high-quality cross-lingual retrieval dataset. 

For the given passage, you must generate a JSON array containing a specific number of query pairs. Each pair must consist of a "Positive Search Query" and a highly challenging "Mined Hard Negative Search Query" written in the target query language.

---

### Part 1: Strict Application & User Persona Framing
The target application is an enterprise search engine sold to law firms. Attorneys use this system to query thousands of dense regulatory documents. 
- **QUERY STYLE:** Queries must be written as **highly concise, dense, professional search strings**. They must mimic an expert attorney trying to immediately surface a specific file by typing in key legal phrases, legal concepts, or highly focused nouns/adjectives. No full sentences, no question marks.

---

### Part 2: Prevent Token Overfitting (Strict Identifier Masking)
To ensure the embedding model learns deep, language-independent legal semantics rather than simple keyword memorization, you must adhere to a strict structural restriction:
- **CRITICAL RULE:** DO NOT include exact statutory numbers, specific regulation codes, article numbers, clause integers, or historical decision years in the queries (e.g., completely ban strings like "EC 761/2001", "Article 11", "97/264/EG").
- **HOW TO ABSTRACT:** Replace exact numbers with their functional legal descriptions or conceptual frameworks. 
  - Instead of "Regulation EC 761/2001", use "EU eco-management and audit scheme framework provisions".
  - Instead of "Article 9", use "statutory cross-border recognition guidelines" or "relevant operational clause".
  - Instead of "Entscheidung 97/264/EG", use "superseded community council directives".

---

### Part 3: Strict Language Code Constraints
All languages in the metadata fields must be identified strictly using their standard lowercase ISO 639-1 two-letter language code: `en`, `de`, or `fr`.

---

### Part 4: Rules for Constructing "Positive Search Queries"
A Positive Search Query is a professional search string that *must uniquely map* to the provided passage as its definitive target based purely on conceptual logic.
1. **Prioritize Semantic Intent over Keyword Overlap:** Focus on the core legal mechanism, compliance thresholds, or underlying enforcement workflows.
2. **Ensure Search Engine Specificity:** Avoid generic search terms (e.g., searching for "environmental auditing" is too broad). Use specific regulatory intersections, environmental vectors, or asset classes so that *only this specific passage* stands out conceptually out of a massive multi-thousand-document corpus.

---

### Part 5: Rules for Constructing "Mined Hard Negatives"
A Hard Negative is a professional search string that looks and smells identical to the passage on a surface semantic level, but is legally mismatched upon close analysis. 
1. **Avoid Contrived or Obvious Negatives:** DO NOT simply negate the positive query using "not" or pivot to an entirely unrelated legal topic.
2. **Apply Semantic Near-Miss Logic:**
   - **Entity/Vector Shifting:** If the passage regulates "soil contamination thresholds for agricultural land", create a hard negative search string about "soil contamination limits for urban residential zones".
   - **Scope/Jurisdiction Blurring:** If the passage specifically governs "inland freshwater fisheries", create a query targeting "deep-sea marine aquaculture".
   - **Conceptual Mimicry:** Write a professional search phrase targeting a highly related legal mechanism, reporting workflow, or compliance timeline that sounds highly plausible within the domain but cannot possibly be resolved by the current passage.

---

### Part 6: Output Formatting Constraints
Your response must be strictly valid JSON, formatted exclusively as a flat array of objects, containing nothing else. Do not wrap the JSON in conversational filler, markdown introductions, or text explanations before or after the code block.

The JSON structure must strictly follow this array schema (ensure `passage_language` and `query_language` contain only `en`, `de`, or `fr`):

```json
[
  {
    "pair_id": 1,
    "passage_language": "en",
    "query_language": "de",
    "positive_query": "[Concise, abstract, professional legal search string containing NO numbers or specific statutory IDs that uniquely targets this passage's semantic logic]",
    "hard_negative": "[A sophisticated, domain-plausible professional search string containing NO numbers or specific statutory IDs that overlaps heavily in topic but is legally mismatched]"
  }
]
```
